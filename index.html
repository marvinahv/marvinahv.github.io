I design and build web applications. I like making things that make the world better. I believe the whole is greater than the sum of its parts. Minimalism and simplicity are awesome.

Programming Skills

I like writing clean and readable code. I like to be DRY and KISS. I love modular stuff. I’m into micro-services. I’m always learning new stuff. I have architected and built web applications and sites from the ground up.

HTML

I advocate for writing semantic html code. This together with a good css architecture make for organized, clean, and re-usable code. It’s hard to believe, but there are still developers who use tables for layouts.

I used haml a lot, until I discovered slim and fell in love with it. I use that as a default in most of my projects. The ability to have modular code with partials and layouts, and the python-like syntax make writing and organizing code a breeze.

When working with backbone.js or other javascript mvcs I have used mustache and other logic-less templates. In particular, I’m pretty fond of hogan.js. I’ve also worked with handlebars. They’re all pretty much the same.

In a project where they were using PHP, I used smarty. But I like Ruby better.

CSS

I have been writing css for a long time and I’ve learnt most of the tricks for making it do what you want it to do (it often doesn’t: ex. vertical-align). I also try to stay current with new features and best practices for it.

Since I discovered sass years ago, I’ve never looked back. I have used less in the past, but in my opinion, sass is way better.

I love architecting my styles in modules and sass is perfect for that. Also, using @mixins, functions, and @extends allow me to have really DRY code. I usually use compass in my sass projects with a couple of compass plugins, especially susy for custom grids. I’ve also used bourbon. Responsive code is much easier with sass, for me.

Frameworks-wise, I’ve used bootstrap in many projects, but I’m not a big fan of it. I think it’s not very semantic and it’s too heavy. I’ve explored several others, but I really like to use pure.css because it’s really minimal but has all the basics and it’s really small. I find it better to just have a foundation and then build upon that with my own design without having to override too much.

JavaScript

I have a lot of experience using JS to manipulate the DOM. I’ve used jQuery extensively, used and written jQuery plugins for a long time.

Now, I usually write coffeescript, but I understand what the underlaying JS code is doing and I could do the same thing with vanilla javascript.

I have used a couple of JavaScript MVC frameworks, but I have the most experience with backbone.js and is one of my favorites, again because of its modularity and non-holistic-framework architecture. I’m currently learning marionette.js for using backbone in bigger applications.

I’m also learning meteor.js and other, more traditional node.js frameworks, such as express.js.

In addition to that, I use mongodb as my default database. I prefer JSON over XML.

Ruby

I love Ruby. I use Ruby to make the simplest programs and it is my tool of choice for web applications.

I originally learnt Ruby to use Rails, but then started using it for task automation, web-scrapping and just everyday scripting.

At first I really enjoyed using Rails, but soon became frustrated with its “magic”. I started using Sinatra and learned to do myself what rails used to do for me. Eventually, I started using Padrino (which is like a mix of sinatra and rails) for bigger projects.

Now I really like the idea of micro-services and I’m using sinatra more and more together with backbone.js.

Databases

I really like mongodb and I use it quite a lot. I love its flexibility. I’ve also used redis and I’m currently learning neo4j.

I’ve also used PostgreSQL for relational databases. In Ruby, I’ve used ActiveRecord with rails and DataMapper in other projects as ORMs. But I can also write my own custom queries when needed.

I believe in using the right tool for the right job.

Web

I also understand the request cycle, the browser’s incompatibilities and differences, and other important issues that must be taken into account for creating great experiences for the users using our products.

I spend a lot of time thinking, prototyping, testing, and improving the features I develop. A lot of the process is not seen (as it can be seen in code), but it is just as important as building it.
